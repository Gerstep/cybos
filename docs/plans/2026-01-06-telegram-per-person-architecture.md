# Telegram Per-Person Conversation Architecture

**Date:** 2026-01-06
**Status:** Planning
**Purpose:** Replace session-based telegram snapshots with persistent per-person conversation files integrated into the entity system.

## Problem Statement

Current architecture creates timestamped snapshots (`YYYY-MM-DD-HH-MM-unread.md`) that:
- Don't provide persistent conversation history per person
- Can't be used for entity context loading
- Don't track sent messages (only incoming at read time)
- Accumulate without providing searchable value

## Proposed Solution

### 1. One File Per Person

```
context/telegram/
├── anton-lobintsev.md      # Persistent conversation log
├── john-smith.md
├── org-team.md             # Groups too
└── .gitkeep
```

**File naming:** Entity slug (same as entity index key)
- "Anton Lobintsev" → `anton-lobintsev.md`
- No date in filename (persistent file)

### 2. Extended Entity Index Schema

**Current schema:**
```json
{
  "entities": {
    "slug": {
      "name": "...",
      "type": "person|org",
      "aliases": [],
      "email": "...",
      "sources": {
        "deal": "...",
        "calls": [],
        "entity_file": null
      }
    }
  }
}
```

**Extended schema:**
```json
{
  "generated": "...",
  "note": "Auto-generated by /cyber-reindex",

  "entities": {
    "anton-lobintsev": {
      "name": "Anton Lobintsev",
      "type": "person",
      "aliases": [],
      "email": "anton@fintech.com",
      "telegram": "lobintsev",              // NEW: username without @
      "sources": {
        "deal": "/deals/antons-fintech",
        "calls": [],
        "entity_file": null,
        "telegram_log": "/context/telegram/anton-lobintsev.md"  // NEW
      }
    }
  },

  "lookups": {                              // NEW SECTION
    "telegram": {
      "lobintsev": "anton-lobintsev"
    },
    "email": {
      "anton@fintech.com": "anton-lobintsev"
    }
  }
}
```

### 3. Conversation File Format

```markdown
# Anton Lobintsev

**Entity:** anton-lobintsev
**Username:** @lobintsev
**Type:** private
**First contact:** 2026-01-06
**Last updated:** 2026-01-06T14:45:00Z
**Last message ID:** 12345

---

## 2026-01-06

- [12:39] **Anton**: Добрый день! Давно читаю Ваш блог про АИ...
- [14:45] **Me**: Привет! Рад что читаете...

## 2026-01-07

- [09:15] **Anton**: Спасибо за ответ...

---
```

**Metadata fields:**
- `Entity:` - links to entity index slug (for cross-reference)
- `Username:` - Telegram handle
- `Type:` - private/group/channel
- `First contact:` - when file was created
- `Last updated:` - when last message was appended
- `Last message ID:` - for deduplication

5. Build `lookups.email` mapping email → entity slug

**Backward compatibility:**
- New fields are optional
- Existing code reads `entities` as before
- Lookups are additive

### Phase 2: Telegram Script Refactor

**File:** `scripts/telegram-gramjs.ts`

**Changes:**

#### 2.1 New functions:

```typescript
// Slug generation from name
function nameToSlug(name: string): string {
  return name
    .toLowerCase()
    .replace(/[^a-z0-9\s-]/g, '')
    .replace(/\s+/g, '-')
    .replace(/-+/g, '-')
    .trim();
}

// Find or create conversation file
function getConversationFilePath(dialog: DialogInfo): string {
  const slug = dialog.username
    ? lookupEntityByTelegram(dialog.username) || nameToSlug(dialog.title)
    : nameToSlug(dialog.title);
  return join(CONTEXT_DIR, `${slug}.md`);
}

// Read existing conversation file metadata
function parseConversationFile(filePath: string): {
  lastMessageId: number;
  lastUpdated: string;
  messages: MessageInfo[];
} | null

// Append new messages to conversation file
function appendToConversationFile(
  filePath: string,
  dialog: DialogInfo,
  newMessages: MessageInfo[]
): void

// Lookup entity by telegram username
function lookupEntityByTelegram(username: string): string | null
```

#### 2.2 Modified flow:

```
CURRENT FLOW:
1. Fetch unread dialogs
2. Read messages
3. Save timestamped scratchpad
4. Save work file
5. Mark unread

NEW FLOW:
1. Fetch unread dialogs
2. For each dialog:
   a. Lookup entity by username in index.lookups.telegram
   b. Get/determine slug (from entity or generate from name)
   c. Find/create context/telegram/<slug>.md
   d. Read existing file, get lastMessageId
   e. Fetch messages (both directions, last 20)
   f. Filter to only NEW messages (id > lastMessageId)
   g. Append new messages to file, update metadata
3. Save work file (for AI drafting - same as before)
4. Mark dialogs unread
5. (Optional) Auto-create entity if new contact
```

#### 2.3 Message capture changes:

- **Both directions:** Capture incoming AND outgoing messages
- **Deduplication:** Skip messages with id <= lastMessageId
- **Next-read capture:** Sent messages appear on next read naturally

### Phase 3: Reindex Integration

**File:** `.claude/commands/cyber-reindex.md`

**Add new source:**

```markdown
### 4. Telegram Logs (`/context/telegram/*.md`)

- Parse file metadata for username
- Extract entity slug from filename or Entity: field
- Type: person (or group for group chats)
- Source link: telegram_log path
- Add to lookups.telegram
```

**Reindex process changes:**

```
CURRENT:
1. Scan deals → orgs
2. Parse calls INDEX → people
3. Scan entity files → merge

NEW:
1. Scan deals → orgs
2. Parse calls INDEX → people
3. Scan entity files → merge
4. Scan telegram files → people/groups   // NEW
5. Build lookups.telegram                 // NEW
6. Build lookups.email                    // NEW
7. Write index
```

### Phase 4: Entity Auto-Creation

When script encounters new Telegram contact not in index:

**Option A: Auto-create (recommended):**
1. Create entity with:
   - name: from Telegram display name
   - type: "person" (or "group")
   - telegram: username
   - sources.telegram_log: path
2. Create conversation file
3. Index updated on next reindex

**Option B: Unlinked file:**
1. Create file with generated slug from name
2. No entity created
3. User can manually link later

**Chosen:** Option A for simplicity

### Phase 5: Remove Old Output Files

**Delete/deprecate:**
- `context/telegram/YYYY-MM-DD-HH-MM-unread.md` pattern
- These session snapshots are replaced by per-person files

**Keep:**
- `content/work/MMDD-telegram-replies-YY.md` - work file for AI drafting

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Break existing entity index | Schema changes are additive, existing fields unchanged |
| Duplicate entities | Use slug-based dedup, same as current |
| Large conversation files | No limit initially, can add rotation later |
| Username changes | Store Telegram ID internally, username is display hint |
| Group chats | Same treatment as private chats, slug from title |
